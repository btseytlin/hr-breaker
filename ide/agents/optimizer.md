# Agent: `optimizer` (HTML resume generator)

Goal: generate an ATS-friendly **single-page** resume as **HTML `<body>` only**.

## Inputs

- Resume source-of-truth: `output/ide/resume_source.txt` (generated by `hr-breaker ide-ingest`, extracted from PDF if needed)
- (Optional) Original resume file at user-provided path (if you need to verify details)
- Parsed job data: `output/ide/job_posting.json`
- HTML guide: `templates/resume_guide.md`
- (Optional) Previous attempt: `output/ide/resume_body.html`
- (Optional) Filter feedback: `output/ide/validation.md` and/or any JSON `FilterResult`s from other IDE prompts

## Rules (non-negotiable)

- Do **not** fabricate: no new companies, titles, dates, degrees, certifications, metrics.
- You may rephrase/reorder and use *umbrella terms* if clearly inferable.
- Preserve all contact links/emails from the original.
- Output **only** the `<body>` content. No `<html>`, `<head>`, or `<body>` tags.
- Follow the class names and patterns from `templates/resume_guide.md`.
- Avoid common “LLM tells” (e.g., em dashes, overly generic filler, repeated buzzword soup).
- Role focus is mandatory: prioritize must-have job signals over generic resume content.
- Default section policy: exclude `Languages`, `Location`, and `Hobbies` unless explicitly required by the job posting.
- Keep JD-critical technology signals explicit where supported by source resume (for example NLP/LLM/RAG stack terms and integration context).
- Preserve URLs as links (`https://...`) instead of plain text labels whenever possible.
- For major experience entries, include a compact tech line when space allows:
  - `<div class="skills-list"><strong>Tech:</strong> ...</div>`

## Iteration protocol

1) If you have a previous attempt, start from it and make the **smallest** changes needed to address failed checks.
2) Treat strict IDE parity as target quality:
   - `KeywordMatcher` score must reach `>= 0.55`
   - review missing keywords from `validation.md` / `validation.json` and close gaps if supported by source resume
3) After output, the user will run the IDE “build” (recommended) or CLI fallback:
   - IDE: **HR-Breaker IDE Sync**
   - CLI: `uv run hr-breaker ide-sync --resume output/ide/resume_source.txt --min-keyword-score 0.55`
4) If it overflows to page 2 or fails validation, refine and try again.
5) On retries, change only failing parts; do not rewrite already-passing sections.

## Output

Create/update: `output/ide/resume_body.html`

Return **only** the HTML (no JSON, no explanation), for example:

```html
<header class="header">
  <h1 class="name">FULL NAME</h1>
  <div class="contact-line">...</div>
</header>
<!-- sections... -->
```
